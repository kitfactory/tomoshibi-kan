# 設計補助ガイド（汎用）

この文書は、プロダクト固有要件に依存しない「設計判断の手順」を定義する。  
目的は、仕様追加時に過剰設計と責務混在を防ぎ、AI/人間のどちらが実装しても一貫した品質を維持すること。

## 0. 位置づけと優先順位

1. 本書は「判断ルール」の正本であり、「機能仕様」の正本ではない。
2. 仕様衝突時は `spec.md` を優先する。
3. レイヤー/依存衝突時は `architecture.md` を優先する。
4. 運用/更新ルールは `OVERVIEW.md` と `AGENTS.md` を優先する。
5. 本書は上記正本を補完し、衝突時の判断を減らすために使う。

## 1. まず固定する判断軸

1. 価値フロー優先: ユーザー価値に直結する流れを切らない。
2. 最小複雑性優先: いま必要な責務とデータのみで成立させる。
3. 責務明確化: 手順実行はUseCase、状態規則はDomainに分離する。
4. 依存方向固定: 外側から内側へ依存し、逆流を許さない。
5. 差し替え可能性: 外部I/Oは境界化し、実装交換を可能にする。
6. 観測可能性: 障害時に追跡可能なイベントと根拠を残す。
7. テスト可能性: 外部依存なしでドメイン規則とユースケースを検証する。

## 2. 設計ルール（汎用）

### 2.1 責務分離
- UseCaseは手順制御と状態遷移だけを担当する。
- Domainは不変条件、整合性、状態表現だけを担当する。
- Adapterは入力/出力の変換だけを担当する。
- InfrastructureはDB/IPC/APIなど技術詳細だけを担当する。

### 2.2 境界と契約
- 外部依存は境界を明示して扱う。
- Interfaceはユースケースが必要とする最小操作に限定する。
- 入力前提条件と出力結果（成功/失敗）を仕様に明記する。
- 例外は仕様化されたエラーへ正規化して返す。

### 2.3 抽象化ゲート（クラス/Port追加条件）
- 次を満たさない限り、抽象化を追加しない。
1. 2つ以上のUseCaseで同一責務が再利用される。
2. 交換予定の実装が現実的に存在する。
3. 追加によってテスト容易性が改善する。
- 上記を満たさない場合は、既存UseCase内で完結させる。

### 2.4 データ設計
- 1データ1責務を守る。
- 必須項目と任意項目を分離する。
- 任意項目を前提に業務処理しない。
- 将来用の先回り属性を追加しない。
- `details/meta` はキー集合を定義し、自由入力領域にしない。

### 2.5 状態遷移・整合性
- 状態は列挙で定義し、許可遷移表を仕様に持つ。
- 不正遷移は副作用なしで失敗させる。
- 再試行/差し戻しは通常系として定義する。
- 同時更新の競合方針（単一ライタ/楽観ロック/排他）を明示する。
- 冪等性が必要な操作は識別子で重複適用を防ぐ。

### 2.6 エラー設計
- エラーは「検知点」「原因」「利用者向け文言」「再試行可否」で定義する。
- 技術エラーと業務エラーを分離する。
- 最低限、入力不備/整合性違反/タイムアウト/依存障害を定義する。

### 2.7 観測性
- 主要イベントに `who / what / target / when / result` を残す。
- 通常表示は要点、監査表示は詳細に分離する。
- 重要判定は根拠（入力、判定結果、制約）を追跡可能にする。

## 3. レイヤー構成ガイド（規模別）

### 3.1 規模別の推奨構成

| 規模 | 判定目安 | 推奨レイヤー | 許容する省略 | 注意点 |
| --- | --- | --- | --- | --- |
| Small (PoC/MVP) | UseCase 1-3、開発者1-2人、外部依存1-2個 | `Core（Domain + UseCase）` + `Boundary（Adapter/Infrastructure）` | AdapterとInfrastructureの同居を許容 | 境界I/Fとエラー正規化は省略しない |
| Medium | UseCase 4-15、継続開発、外部依存複数 | `Domain` / `UseCase` / `Interface Adapter` / `Infrastructure` | PresenterとControllerの統合は可 | UseCaseへの技術依存混入を防ぐ |
| Large (Multi-team) | UseCase 16以上、複数チーム、監査/可用性要件あり | 4層 + 機能モジュール分割（境界を明文化） | 省略より分割を優先 | 依存逆流検知と契約テストを必須化 |

### 3.1.1 規模閾値の調整ルール

- 上表のUseCase数/人数/依存数は既定値（初期値）として扱う。
- プロジェクト事情で乖離がある場合は `architecture.md` に「規模判定基準」を定義し、上表を上書きしてよい。
- 上書き時は「採用理由」「適用日」「見直し条件（例: 四半期、UseCase +3）」を記録する。
- 判定が境界にある場合は、過小設計を避けるため上位規模を採用する。

### 3.2 層を増やす/分けるトリガー

- 同一責務の重複が2箇所以上で発生した。
- UseCaseがI/O詳細やUI都合を直接持ち始めた。
- 障害時に「どの層の問題か」を即時切り分けできない。
- 実装差し替え（DB/LLM/API）が現実的に発生した。
- テストでモック境界が曖昧になり、失敗要因が追跡しづらい。

### 3.3 本プロジェクト適用例（PalPal-Hive）

| レイヤー | 置くもの | 置かないもの |
| --- | --- | --- |
| Domain | `Plan`, `Task`, `Pal`, `CompletionRitual`, `GateResult`, `Event`, `UserMessage`, `GuideMessage`, 各Status型（UI表示名: Plan Card / Task Card） | DB/IPC呼び出し、LLM接続 |
| UseCase | `GuideConversationUseCase`, `GuideDispatchUseCase`, `PalWorkUseCase`, `GateReviewUseCase`, `PalResubmitUseCase`, `GuideCompletionUseCase` | UI制御詳細、SQL文 |
| Interface Adapter | `AppController`, `WorkspacePresenter` | 業務ルール本体 |
| Infrastructure | `SqliteWorkflowRepository`, `LlmRuntime`, `PalpalCoreGateJudge`, `ElectronIpcRouter`, `GuideGateProfileLoader` | ドメイン判断の本体 |

### 3.4 本プロジェクトの体験トーン（PalPal-Hive）

- 詳細な体験トーンと視覚判断は `experience_tone.md` を正本とする。
- トーン定義: `温かみと知性` を両立し、「のびのび仕事・研究できるが、判断は甘く見えない」空気を優先する。
- 基本姿勢: `cute` より `thoughtful`、`cozy` より `calm`、`friendly` より `reassuring` を選ぶ。
- 画面別トーン配分:
  - `Guide Chat` / `Project` / `Settings`: 親和的、安心感、探索しやすさを優先する。
  - `Task Board` / `Gate` / `Event Log`: 厳密さ、静けさ、根拠の見やすさを優先する。
- 色の原則: 暖色は残してよいが、高彩度でポップにせず、紙・木・アンバー寄りの落ち着いた温度で使う。
- 面の原則: すべてを同じ柔らかさにせず、会話面はやや柔らかく、判定面は輪郭と整列を強める。
- 動きの原則: 常時動く装飾より、入力・保存・送信・判定など状態変化に連動する静かな反応を優先する。
- キャラクターの原則: Guide表現は装飾物ではなく、送信中・成功・エラーなどの状態を補助する役割で使う。
- 情報密度の原則: 知性は装飾よりも、余白、整列、ラベル精度、優先順位の明快さで表現する。
- 避ける表現: 過度にポップ、幼い、未来感を誇張した青白さ、監査面まで軽く見える演出。

#### 3.4.1 画面ごとのトーン配分

| 画面 | 主トーン | 副トーン | ねらい |
| --- | --- | --- | --- |
| `Guide Chat` | 親和的 | 集中 | 相談を始めやすくしつつ、会話の思考密度は保つ |
| `Project` | 探索的 | 安心感 | 参照対象を広げても散らからない印象を保つ |
| `Settings` | 整理された安心感 | 実務性 | 設定変更時の不安を減らし、理解と保存判断を支える |
| `Pal List` | 親しみ | 構造感 | エージェントの人格を見せつつ、役割差を読み取りやすくする |
| `Task Board` | 静かな厳密さ | 前進感 | 何を今やるべきかを明瞭に見せる |
| `Gate` | 厳格さ | 公正さ | 判定の重みと理由の明快さを優先する |
| `Event Log` | 観測可能性 | 静けさ | 監査情報を冷静に追える表示にする |
| `Cron` | 安定感 | 規律 | 定期運用の反復性と信頼感を出す |

## 4. AI/実装者の判断プロトコル

1. 要求を「仕様追加」「設計変更」「実装修正」に分類する。
2. 該当する正本（spec/architecture/overview）を先に確認する。
3. 変更案を最小単位で作る（不要属性・不要クラスを増やさない）。
4. 境界責務と依存方向をセルフレビューする。
5. テストとログで検証し、結果を文書化してから完了とする。

## 5. 実装時の境界チェック

1. UseCaseにSQL/HTTP/IPCが漏れていないか。
2. Domainに外部依存呼び出しが入っていないか。
3. Adapterに業務判断の本体が入っていないか。
4. Infrastructureが業務ルールを持っていないか。
5. 主要UseCaseごとに入力/出力/失敗条件が定義済みか。
6. 競合時の整合性（同時実行、再実行）が明記されているか。
7. 選択したレイヤー数が現在規模に対して過剰/過小になっていないか。

## 6. NGパターン（再発防止）

- 依存方向の逆流。
- 1クラスへの責務集中（God Class / God API）。
- 用途不明属性の先行追加。
- 失敗条件を例外に丸投げして仕様化しない。
- 判定根拠を記録しない。
- 抽象化のための抽象化を行う。

## 7. テスト最小基準

1. Domain: 不変条件/遷移制約の単体テストを持つ。
2. UseCase: 正常系1本 + 主要失敗系1本以上を持つ。
3. Adapter/Infra: 契約テストまたは結合テストでI/F整合を確認する。
4. 回帰防止: 変更対象に対応する自動テストを最低1件追加する。

## 8. 変更前チェック（10項目）

1. この変更は価値フローに直接寄与するか。
2. 既存責務と既存データで実現できないか。
3. 依存方向とレイヤー責務を維持できているか。
4. 新規Interface/DTO/状態は最小か。
5. 選択したレイヤー構成は現在規模（UseCase数・チーム数・依存数）に見合うか。
6. エラー時の再試行・復旧手順は定義済みか。
7. ログで原因追跡できるか。
8. 競合・再実行時の整合性は保証されるか。
9. テストで回帰防止できるか。
10. 正本（spec/architecture/overview）との矛盾はないか。

この10項目で1つでも `No` があれば、設計を分解してから実装する。

```

- 外部依存（DB/API/IPC/LLMなど）:
- 期待するユーザー価値（価値フロー）:

4. レイヤー配置:
   - Domain: 追加/変更する型・不変条件
   - UseCase: 手順・状態遷移・入力/出力
   - Adapter: 変換のみ（UI/外部I/O）

   - 規模判定基準は本書既定値を使い、`architecture.md` に上書き定義があればそちらを優先

   - Medium/Large: Domain / UseCase / Interface Adapter / Infrastructure

5. 境界と契約（Port最小操作、入力前提、成功/失敗出力、例外→仕様化エラー）
6. 状態遷移・整合性（状態一覧、許可遷移、競合方針、冪等性）
7. エラー設計（入力不備/整合性違反/タイムアウト/依存障害 + 検知点・原因・利用者文言・再試行可否）

9. テスト最小基準（Domain単体、UseCase:正常1+主要失敗1以上、Adapter/Infra:契約or結合）
10. 境界チェック（Yes/No）:
    - UseCaseにSQL/HTTP/IPCが漏れていないか
    - Domainに外部依存呼び出しが入っていないか
    - Adapterに業務判断の本体が入っていないか
    - Infrastructureが業務ルールを持っていないか
    - 主要UseCaseごとに入力/出力/失敗条件が定義済みか
    - 競合時の整合性（同時実行、再実行）が明記されているか
    - 選択したレイヤー数が現在規模に対して過剰/過小になっていないか
11. 変更前チェック10項目（Yes/No）:
    - この変更は価値フローに直接寄与するか
    - 既存責務と既存データで実現できないか
    - 依存方向とレイヤー責務を維持できているか
    - 新規Interface/DTO/状態は最小か
    - 選択したレイヤー構成は現在規模（UseCase数・チーム数・依存数）に見合うか
    - エラー時の再試行・復旧手順は定義済みか
    - ログで原因追跡できるか
    - 競合・再実行時の整合性は保証されるか
    - テストで回帰防止できるか
    - 正本（spec/architecture/overview）との矛盾はないか
    - 原則: Noが1つでもあれば分解案を提示して終了
    - 例外: 実装修正かつ局所変更（既存I/F・状態・データモデル不変）は改善案提示のうえ継続可
12. 実装タスク分解（ファイル/クラス単位の箇条書き）
```
