# ui_tone_application.md（UI修正方針）

## 1. 位置づけ

- この文書は `experience_tone.md` を UI 修正へ落とすための実行ガイドである。
- 体験トーンの正本は `experience_tone.md` とする。
- 本書は「どの画面に、どの方向の修正を、どの優先度で入れるか」を定義する。

## 2. 実装優先度

### P1: まず揃える
- 面の階層を `主役 / 補助 / 操作` の 3 段階へ整理する。
- タブ、カード、モーダル、トーストの影とグラデーションを減らし、質感差を明確にする。
- マイクロインタラクションの時間と反応方向を統一する。
- Guide キャラクターを常時装飾から状態連動へ寄せる。

### P2: 次に効かせる
- `Guide Chat` と `Settings` の安心感を、余白と保存フィードバックで強める。
- `Task Board` / `Gate` / `Event Log` の厳密さを、ラベル精度と情報階層で強める。
- ステータス色を彩度依存から明度差・コントラスト依存へ移す。

### P3: 最後に詰める
- 画面間でタイポグラフィと行間のルールを統一する。
- 空状態、成功状態、失敗状態の演出をトーンに合わせて統一する。
- 反復利用時の疲れを減らすため、背景演出と浮遊要素の主張を最適化する。

## 3. 共通修正方針

### 3.1 色
- ベースは暖色寄りの中立色を維持する。
- 強調色は 1 系統に寄せ、画面ごとに色の役割を増やしすぎない。
- エラーや拒否は派手な赤ではなく、厳格さが伝わる深い警告色で扱う。

### 3.2 面
- 主要コンテナは「静かな一枚板」の印象を優先し、過剰な立体感を避ける。
- カードの差は影量よりも、輪郭、余白、背景濃度で出す。
- モーダルは特別扱いするが、華美にせず「判断の場」として締める。

### 3.3 動き
- hover は浮くよりも「少し整う」方向へ寄せる。
- 保存、送信、承認、拒否は終わりが明確に分かる短い反応を入れる。
- 常時アニメーションは減らし、状態遷移時だけ意味のある動きにする。

### 3.4 タイポグラフィ
- 見出しは柔らかさよりも、構造が見える強さを優先する。
- ラベルは短く、実装用語をそのまま出しすぎない。
- 会話文と監査文で同じ組版リズムを使わない。

## 4. 画面別修正方針

### 4.1 Guide Chat
- 入力欄は「安心して話しかけられる」ことを優先し、柔らかさを維持する。
- ただし bubble や header の密度は整え、思考の場として軽く見えないようにする。
- Guide キャラクターは常時表示ではなく、送信中、応答完了、エラーなどにだけ反応させる。

### 4.2 Settings
- 現状の分かりやすさは維持し、見た目の静けさを上げる。
- 保存フッターはそのまま活かし、成功時のフィードバックを強める。
- モデル追加、スキル追加は「管理」より「安心して設定できる」印象へ寄せる。

### 4.3 Task Board
- タイトル、状態、次アクションが最初の視線で読めるようにする。
- 内部情報は 1 段下げ、実務に必要な項目だけを強く見せる。
- カードの hover は派手な浮上ではなく、選択性の強化に使う。

### 4.4 Gate
- 最も厳格な画面として扱う。
- 理由入力、Approve/Reject、テンプレート選択は、遊びを抑え、判断の重みを優先する。
- 成功よりも「判断した」ことが明確に伝わる終わり方にする。

### 4.5 Event Log
- 装飾を減らし、監査ログとしての読みやすさを優先する。
- 情報密度は高くてよいが、構造は冷静であるべき。
- 一覧カードより、タイムラインまたはログビューに寄せる方向を優先する。

### 4.6 Project / Pal List / Cron
- `Project` は探索感を保ちつつ、散らかって見えない整理感を強める。
- `Pal List` は人格表現を残しつつ、役割差の読み取りやすさを優先する。
- `Cron` は定期運用らしい安定感と規律を優先し、飾りすぎない。

## 5. 最初の実装スライス

1. カラートークンの再整理
2. 面の階層整理
3. マイクロインタラクション時間の統一
4. Guide キャラクターの状態連動化
5. `Task Board` / `Gate` / `Event Log` の見た目を厳密側へ寄せる

## 6. 非対象

- 新機能追加
- 情報設計の大規模変更
- タブ構成の変更
- 3D 演出や大きな背景演出の追加
